# Playwright RTM Generator

Produces a per-lab Requirements Traceability Matrix: AC → test case → type → result → defect.
This is the STLC artifact that distinguishes traceable test design from "some Playwright tests".

---

## ⛔ Guardrail — Lab or JIRA Key Required

Scan the prompt for a JIRA key (`TAB1-\d+`) or a lab name. Resolve to the spec path
`tests/<lab-name>/<lab-name>.spec.ts`.

**If the spec does not exist → STOP:**

> ⛔ No spec found for `<lab-name>`. Run `test-case-generator` first, then re-run this skill.

---

## Phase 1 — Collect Acceptance Criteria

- If Atlassian MCP is available, fetch `TAB1-XX` and extract the AC bullets (numbered AC-1…AC-N).
- Otherwise, parse the `// AC-N (TAB1-XX): …` comments already present in the spec file.
- Always include the implicit requirement: **AXE — the page must have no critical axe violations.**

---

## Phase 2 — Collect Test Cases from the Spec

Read `tests/<lab-name>/<lab-name>.spec.ts` and walk its structure:

- Each `test.describe('AC-N — …')` group → the AC it traces to.
- Each `test('…')` inside it (including those emitted in `for…of` data-driven loops) → one RTM row.
- Tag each case by type: **P** positive · **N** negative · **B** boundary · **D** data-driven · **A11y**.
- Note any `test.describe` not tied to an AC (e.g. "Optional controls") under a `—` requirement.

---

## Phase 3 — Attach Results & Defects

- If `playwright-report/results.json` exists, read each test's `status` and fill the Result
  column (✅ passed / ❌ failed / ⏭ skipped). Otherwise mark Result as `—` (not yet run).
- For any test that intentionally filters a known axe violation (a `v.id !== '…'` filter with a
  defect comment), create a **Defect** row: id, severity (from axe `impact`), summary, found-by
  test, JIRA key, status. Cross-link it from the AXE row with a `*` footnote.

---

## Phase 4 — Write the RTM

Write `docs/rtm/<lab-name>.rtm.md` using this structure (header table, coverage table grouped
by AC, a Defects table, and a traceability summary):

```
# Requirements Traceability Matrix — <Lab Name>

| Field | Value |  (JIRA story, Lab URL, Spec, POM, Last run, Generated by)

## Coverage by Acceptance Criterion
| Req | Acceptance Criterion | Test case | Type | Result |

## Defects
| ID | Severity | Summary | Found by | JIRA | Status |

## Traceability summary
- ACs covered: N / N
- Test cases: M (P / N / B / D / A11y)
- Every POM error locator asserted by ≥1 negative case: yes/no
- Open defects: K
```

**Coverage rule to verify:** every error/validation locator declared in the POM must appear in
at least one negative case. If any is unreferenced, flag the AC as under-covered in the summary.

---

## Phase 5 — Summary

```
✅ RTM written: docs/rtm/<lab-name>.rtm.md
   ├─ ACs: N / N covered
   ├─ Cases: M  (P:x N:y B:z D:w A11y:v)
   ├─ Defects: K open
   └─ Under-covered ACs: <list or "none">
```

---

## Quick-Start Prompt Template

```
1. Resolve lab/JIRA key → spec path
2. Collect ACs (Atlassian MCP or // AC-N comments) + the always-on AXE requirement
3. Walk the spec: describe-per-AC → test cases, tagged P/N/B/D/A11y
4. Merge results.json statuses + extract tracked-defect filters into Defect rows
5. Write docs/rtm/<lab-name>.rtm.md
6. Print summary with coverage + open-defect counts
```
